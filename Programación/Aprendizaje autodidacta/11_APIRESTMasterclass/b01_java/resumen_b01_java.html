    Ruta de dominio Java moderno

    Este bloque se domina cuando dejas de ver las herramientas como sintaxis nueva y empiezas a elegirlas por el problema que eliminan: mutabilidad accidental, nulls, duplicacion, colecciones verbosas, errores sin modelar y concurrencia insegura.

    Modelar valores

    Usa records e igualdad correcta para que los datos sean pequenos, claros e inmutables.

    Expresar ausencia

    Usa Optional como contrato de retorno y evita nulls silenciosos que explotan lejos.

    Transformar colecciones

    Usa streams, collectors y lambdas para leer la intencion antes que el mecanismo.

    Cerrar casos raros

    Usa sealed, excepciones con causa, java.time y atomicos para reducir estados imposibles.

    Matriz de eleccion

    Solo transporta datos

    Record. Si hay listas, copia defensiva para inmutabilidad real.

    Puede no existir

    Optional como retorno; orElseThrow si la ausencia es error de negocio.

    Lista a informe

    Stream + collector. Si agrupas y sumas, usa downstream en vez de mapas manuales.

    Casos cerrados

    Sealed + switch exhaustivo. Evita booleanos y strings magicos.

    Senales de dominio

    Puedes explicar PECS, por que orElseGet es perezoso, por que contador++ no es atomico y por que equals/hashCode son inseparables.

    Cuando volver atras

    Si aparece Optional.get(), streams con mutacion externa, catch(Exception) vacio o listas genericas que no compilan, vuelve a esta ruta.

    1. Records: DTOs inmutables

    Un record es una clase de datos compacta. El compilador genera constructor, accesores, equals, hashCode y toString.

    public record UsuarioDto(Long id, String nombre, String email) {
    public UsuarioDto {
        if (email == null || email.isBlank()) {
            throw new IllegalArgumentException("email obligatorio");
        }
        nombre = nombre.trim();
    }

    public UsuarioDto conNombre(String nuevoNombre) {
        return new UsuarioDto(id, nuevoNombre, email);
    }
}

    Idea clave

    Inmutable significa que no se modifica: se crea otro valor. Esto reduce bugs de estado y facilita compartir datos entre hilos.

    Un record puede tener validacion y metodos derivados, pero no estado adicional mutable.

    Popular

    DTOs de entrada/salida, claves de mapas, respuestas de servicio, valores de dominio pequenos.

    No tan popular

    Constructor canonico explicito cuando necesitas anotaciones por parametro o normalizacion mas controlada.

    Cuidado

    Los componentes son referencias finales, pero si guardas una lista mutable dentro, el contenido de esa lista todavia puede cambiar.

    2. Optional: ausencia como parte del tipo

    Un metodo que devuelve Optional<T> declara que puede no haber resultado. El consumidor debe decidir que hacer.

    MetodoUsoTrampa
    ofValor obligatorio.Lanza NPE si recibe null.
    ofNullableFrontera con codigo legacy.No lo uses para esconder diseno confuso.
    mapTransforma si existe.Si la funcion devuelve Optional, genera anidamiento.
    flatMapTransforma evitando Optional<Optional<T>>.La funcion debe devolver Optional.
    filterVaciar si no cumple condicion.No reemplaza validaciones de negocio complejas.
    orElseDefault ya calculado.Se evalua aunque haya valor.
    orElseGetDefault perezoso.Ideal para calculos caros.
    orElseThrowAusencia como excepcion.Patron natural para 404.

    Patron API

    return repo.findById(id)
    .map(UsuarioDto::from)
    .orElseThrow(() -> new NotFound(id));

    Antipatron

    isPresent() + get() suele ser un null-check disfrazado. Encadena operaciones.

    Regla

    Optional como retorno, no como campo ni parametro. Es convencion del JDK y evita modelos raros.

    3. Streams y collectors

    Un stream es una tuberia perezosa: operaciones intermedias preparan el trabajo; una terminal lo ejecuta.

    List<Pedido>

    Datos de entrada.

    stream()

    Abre la tuberia.

    filter/map

    Transforma sin mutar.

    collect

    Operacion terminal.

    Resultado

    Lista, mapa, total, boolean...

    OperacionTipoUso
    filterIntermediaConservar elementos que cumplen predicado.
    mapIntermediaTransformar 1 a 1.
    flatMapIntermediaAplanar 1 a muchos.
    sorted/distinctIntermediaOrdenar o quitar duplicados.
    toList/count/sumTerminalMaterializar o agregar.
    anyMatch/findFirstTerminalComprobar o buscar.

    Collectors

    groupingBy agrupa, partitioningBy separa true/false, joining concatena, toMap convierte a mapa.

    Si hay claves duplicadas en toMap, define funcion de fusion.

    Detalle fino

    Stream.toList() devuelve lista inmodificable. Si vas a usar add, crea un ArrayList.

    Orden

    groupingBy usa normalmente HashMap. Si el orden importa, pide LinkedHashMap.

    Paralelo

    parallelStream() no es turbo gratis. Requiere operaciones independientes, sin estado compartido y coste suficiente.

    4. Genericos, wildcards y PECS

    Los genericos permiten escribir repositorios, servicios y utilidades reutilizables sin perder tipo. Los wildcards expresan varianza segura.

    public class InMemoryRepository<T, ID> {
    private final Map<ID, T> data = new HashMap<>();
    private final Function<T, ID> idExtractor;

    public T save(T entity) {
        data.put(idExtractor.apply(entity), entity);
        return entity;
    }

    public Optional<T> findById(ID id) {
        return Optional.ofNullable(data.get(id));
    }
}

    Erasure

    En runtime se borran los parametros de tipo: List<String> y List<Integer> comparten clase. Por eso no puedes hacer new T().

    ? extends T

    Productor: lees como T, no anades porque desconoces el subtipo exacto.

    ? super T

    Consumidor: puedes anadir T, al leer obtienes algo tan general como Object.

    PECS

    Producer Extends, Consumer Super. Es la regla que desbloquea firmas como Collections.copy.

    NecesidadFirma mentalEjemplo
    Leer animales de cualquier subtipoList<? extends Animal>Calcula peso total.
    Meter perros en una lista compatibleList<? super Perro>Rellenar destino con perros.
    Ordenar elementos comparables<T extends Comparable<T>>Buscar minimo/maximo.

    5. Interfaces funcionales y lambdas

    Una interfaz funcional tiene un unico metodo abstracto. Una lambda permite pasar comportamiento sin crear una clase entera.

    InterfazFirmaSe lee comoEjemplo
    Predicate<T>T -> booleancumple?u -> u.edad() >= 18
    Function<T,R>T -> RtransformaUsuario::nombre
    Consumer<T>T -> voidefectoSystem.out::println
    Supplier<T>() -> TfabricaArrayList::new
     
    Predicate<Usuario> adulto = u -> u.edad() >= 18;
Predicate<Usuario> activo = Usuario::activo;
Predicate<Usuario> clienteValido = adulto.and(activo);

Function<String, String> normalizar =
    String::trim
        .andThen(String::toLowerCase);

    Referencias ::

    Estaticas, de instancia sobre parametro, de instancia sobre objeto concreto y de constructor. Mejoran lectura cuando el nombre del metodo dice suficiente.

    6. Sealed, pattern matching y excepciones

    Sealed cierra una jerarquia y permite switches exhaustivos. Las excepciones bien hechas conservan causa y expresan negocio.

    public sealed interface Resultado permits Ok, NotFound, Error {}
public record Ok(Object dato) implements Resultado {}
public record NotFound(String id) implements Resultado {}
public record Error(String motivo) implements Resultado {}

int status(Resultado r) {
    return switch (r) {
        case Ok ok -> 200;
        case NotFound nf -> 404;
        case Error err -> 500;
    };
}

    Ventaja

    Si anades otro subtipo, el compilador senala cada switch incompleto. No necesitas default que oculte olvidos.

    Try-with-resources

    Todo AutoCloseable se cierra aunque haya excepcion. Si hay varios recursos, se cierran en orden inverso.

    Causas

    Al envolver excepciones, conserva la causa: new RepoException("fallo", e). Sin causa pierdes la pista real.

    Checked vs unchecked

    En APIs modernas, las excepciones de negocio suelen ser unchecked y luego se traducen a HTTP en una capa central.

    7. java.time y concurrencia minima viable

    Las APIs reales manejan fechas, zonas, expiraciones y muchas peticiones simultaneas. Estos temas son pequenos hasta que fallan.

    ClaseRepresentaUso API
    LocalDateFecha sin hora.Cumpleanos, vencimientos de calendario.
    LocalDateTimeFecha y hora sin zona.Solo si la zona no importa o se conoce fuera.
    InstantPunto exacto UTC.Persistencia y JSON tecnico.
    ZonedDateTimeFecha/hora con zona.Eventos humanos en zona concreta.
    DurationTiempo maquina.Timeouts, expiraciones, latencia.
    PeriodTiempo calendario.Meses, anos, reglas humanas.

    Regla de oro

    Guarda instantes en UTC; presenta en la zona del usuario. La zona horaria es presentacion salvo que el dominio diga lo contrario.

    Atomicos

    contador++ no es atomico. Usa AtomicInteger.incrementAndGet() para contadores compartidos.

    ConcurrentHashMap

    merge permite sumar por clave de forma atomica. No acepta claves ni valores null.

    CompletableFuture

    supplyAsync lanza trabajo, thenApply transforma, thenCombine junta resultados, join espera.

    CompletableFuture<Precio> precio = CompletableFuture.supplyAsync(() -> consultarPrecio(id));
CompletableFuture<Stock> stock = CompletableFuture.supplyAsync(() -> consultarStock(id));

Ficha ficha = precio.thenCombine(stock, Ficha::new).join();

    8. Contratos equals/hashCode

    Si dos objetos son iguales por equals, deben tener el mismo hashCode. Romperlo destruye colecciones hash.

    @Override
public boolean equals(Object o) {
    if (this == o) return true;
    if (!(o instanceof Usuario u)) return false;
    return Objects.equals(id, u.id);
}

@Override
public int hashCode() {
    return Objects.hash(id);
}

    Identidad vs valor

    Entidades con BD suelen comparar por id estable. Objetos de valor comparan por todos sus campos, y por eso un record suele ser ideal.

    Sintoma clasico

    Metes un objeto en un HashSet y contains devuelve false para otro "igual". El hash lo mando al cajon equivocado.

    Mutabilidad

    No uses campos mutables en hashCode si el objeto vive dentro de un HashSet/HashMap. Cambiar el campo cambia su posicion logica.

    Consistencia

    Los campos usados en equals y hashCode deben coincidir. Si cambias uno, cambia el otro.
