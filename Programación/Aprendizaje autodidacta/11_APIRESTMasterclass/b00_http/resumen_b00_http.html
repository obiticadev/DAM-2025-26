    Resumen expandido

    HTTP antes de Spring: la gramatica real de una API REST

    Una API no empieza en un controlador, empieza en una peticion con metodo, ruta, cabeceras, cuerpo y una respuesta coherente. Este fichero concentra la teoria del bloque, la conecta con los ejercicios y anade retos frecuentes y menos frecuentes para cubrir el contenido con margen.

    El bloque 00 forma el suelo: parsear mensajes, elegir codigos, respetar verbos, entender cabeceras, negociar formatos, disenar recursos y aprovechar cache sin romper la naturaleza stateless.

    Ej010

    Ruta de dominio HTTP

    La forma mas rentable de estudiar este bloque es pasar de lo mecanico a lo arquitectonico: primero lees mensajes, luego decides semantica, despues modelas recursos y por ultimo optimizas el intercambio.

    Leer el cable

    Reconoce start-line, headers, body, CRLF, Content-Length y limites del mensaje sin depender de herramientas visuales.

    Nombrar la intencion

    Elige metodo y status por significado, no por costumbre. Aqui se gana consistencia.

    Disenar recursos

    Convierte acciones de negocio en sustantivos, colecciones, elementos, filtros y subrecursos.

    Reducir friccion

    Usa cache, negociacion, statelessness y cabeceras para hacer la API mas barata y escalable.

    Matriz de decisiones rapidas

    No entiendo la request

    Usa 400. Ejemplos: JSON roto, query ilegible, Content-Length ambiguo.

    Entiendo, pero no cumple negocio

    Usa 422 o 409 segun sea validacion o conflicto con estado existente.

    El cliente ya tiene copia

    Usa ETag + If-None-Match y responde 304 sin body si coincide.

    La accion no es CRUD puro

    Modelala como subrecurso: POST /pedidos/7/cancelacion.

    Senales de dominio

    Puedes explicar por que DELETE es idempotente, por que 401 necesita WWW-Authenticate, por que 204 no lleva body y por que Accept no es Content-Type.

    Cuando volver atras

    Si dudas entre 400/422/409, si tus rutas tienen verbos, o si un test depende de mayusculas exactas en headers, vuelve a esta seccion antes de seguir.

    1. HTTP en bruto

    HTTP es un protocolo peticion-respuesta. En HTTP/1.1 se puede leer como texto; en HTTP/2 y HTTP/3 viaja en frames binarios, pero la semantica de metodos, codigos y cabeceras sigue siendo la misma.

    GET /usuarios/42?expand=roles HTTP/1.1
Host: api.demo.local
Accept: application/json
Authorization: Bearer eyJ...


    HTTP/1.1 200 OK
Content-Type: application/json
ETag: "u42-v3"
Cache-Control: no-cache

{"id":42,"nombre":"Ana","roles":["ADMIN"]}

    CRLF y LF

    El estandar usa \r\n, pero un parser didactico robusto tolera \n. La tecnica practica es separar por salto de linea y limpiar el \r residual.

    Linea en blanco

    Es la frontera entre metadatos y cuerpo. Antes: metodo/ruta/version y headers. Despues: body.

    Body seguro

    Content-Length delimita bytes exactos. Si aparecen longitudes contradictorias, no se adivina: se rechaza con 400.

    2. Anatomia de request y response

    La peticion pregunta o manda una intencion; la respuesta comunica resultado, metadatos y opcionalmente una representacion.

    Request

    ParteFuncionTrampa habitual
    MetodoGET, POST, PUT...Inventar verbos en la URL: /crearUsuario.
    RutaIdentifica recurso y puede incluir query.Decodificar %26 antes de separar por &.
    VersionHTTP/1.1, HTTP/2.Parsear asumiendo una unica version.
    HeadersMetadatos case-insensitive.Cortar por todos los dos puntos.
    BodyDatos enviados al servidor.Ignorar Content-Length.

    Response

    La primera linea cambia: version, codigo y frase. La frase es para humanos y en HTTP/2 desaparece, asi que las maquinas deben mirar el numero.

    Reglas de oro: 201 suele llevar Location; 204 y 304 no llevan body; si mandas JSON, declara Content-Type: application/json.

    3. Codigos de estado

    El primer digito cuenta la historia: 2xx exito, 3xx redireccion/cache, 4xx problema del cliente, 5xx problema del servidor.

    CodigoNombreUso correctoMatiz fino
    200OKLectura o actualizacion con cuerpo.No lo uses para esconder errores.
    201CreatedRecurso creado.Anade Location.
    204No ContentExito sin representacion.Nunca body.
    304Not ModifiedCache valida.Nunca body; el cliente reutiliza copia.
    400Bad RequestSintaxis ilegible o request malformada.JSON roto, parametro no parseable, CL ambiguo.
    401UnauthorizedFalta autenticacion o no vale.Debe indicar WWW-Authenticate.
    403ForbiddenSe quien eres, pero no puedes.Reintentar igual no ayuda.
    404Not FoundRecurso inexistente.A veces se usa para no filtrar existencia.
    409ConflictChoca con el estado actual.Email duplicado, version obsoleta.
    422Unprocessable EntitySintaxis valida, negocio invalido.Si tu API usa 400 para todo, se consistente.
    429Too Many RequestsRate limiting.Acompana con Retry-After si puedes.
    500Internal Server ErrorBug o excepcion no controlada.No expongas trazas al cliente.
    503Service UnavailableSobrecarga o mantenimiento.Puede ser temporal.

    4. Metodos, seguridad e idempotencia

    La propiedad clave no es si devuelve lo mismo, sino si deja el servidor en el mismo estado al repetir la operacion.

    MetodoIntencionSeguroIdempotenteBody tipico
    GETLeer recurso.SiSiNo
    HEADLeer solo headers.SiSiNo
    OPTIONSDescubrir operaciones/CORS.SiSiNo
    POSTCrear o ejecutar accion no idempotente.NoNoSi
    PUTReemplazar recurso completo.NoSiSi
    PATCHModificar parcialmente.NoNormalmente noSi
    DELETEEliminar.NoSiNo

    Seguro

    No modifica estado. Un GET que borra datos rompe navegadores, crawlers, caches y cualquier herramienta que precargue enlaces.

    Idempotente

    Un DELETE puede devolver 204 la primera vez y 404 la segunda, pero el estado final sigue siendo "recurso borrado".

    Reintentos

    PUT y DELETE pueden reintentarse mejor que POST. Por eso pasarelas y clientes miran la semantica antes de repetir.

    5. Headers y negociacion de contenido

    Los headers son el contrato lateral de HTTP. Indican formato, cache, credenciales, longitud, localizacion, preferencias y seguridad.

    Catalogo esencial

    HeaderDireccionIdea
    HostRequestDominio destino, obligatorio en HTTP/1.1.
    AcceptRequestLo que quiero recibir.
    Content-TypeAmbasLo que estoy enviando.
    AuthorizationRequestCredenciales: Bearer, Basic...
    LocationResponseURI del recurso creado o redireccion.
    ETagResponseHuella/version del recurso.
    If-None-MatchRequestRevalidacion de cache.
    Strict-Transport-SecurityResponseFuerza HTTPS en navegadores.

    Reglas de parsing

    Los nombres son case-insensitive; los valores conservan su caso. Se corta por el primer :, porque Host: localhost:8080 es valido.

    Accept mira al futuro; Content-Type describe el presente.

    Accept con q

    Accept: application/json, application/xml;q=0.8, */*;q=0.1

    Se elige el formato soportado con mayor calidad. Si ninguno encaja, la respuesta correcta es 406 Not Acceptable. Los parametros como charset=utf-8 no cambian el tipo base al comparar.

    6. URLs, query strings y percent-encoding

    La URL no es solo texto bonito. Cada parte tiene responsabilidad y algunas ni siquiera llegan al servidor.

    https://api.tienda.com:443/v1/productos?categoria=libros&tag=java&tag=http#detalle
└─ esquema             └ host ┘ puerto └ path ┘ └──── query string ────┘ fragmento

    Path

    Identifica recursos y jerarquia: /usuarios/42/pedidos. No debe almacenar filtros variables.

    Query

    Filtros, orden, paginacion, busqueda y banderas: ?page=2&sort=precio. Una clave puede repetirse.

    Fragmento

    #detalle se queda en el navegador. El servidor no lo ve, asi que no lo uses para logica backend.

    Percent-encoding

    Separa primero por & y =, decodifica despues. Si decodificas antes, %26 se convierte en un separador falso.

    Valores vacios

    ?debug puede significar bandera presente; ?q= clave con valor vacio. Decide y documenta.

    Mas alla del bloque

    Normaliza barras, evita IDs sensibles predecibles cuando importe, y no metas estado de sesion en query.

    7. Modelado REST y madurez de Richardson

    REST no es "usar JSON". Es nombrar recursos con URIs, aplicar metodos HTTP con semantica y devolver codigos adecuados.

    OperacionRuta RESTExitoFallo habitual
    ListarGET /productos200 + arrayNormalmente no 404 por lista vacia.
    DetalleGET /productos/{id}200404
    CrearPOST /productos201 + Location400/422
    ReemplazarPUT /productos/{id}200 o 204404/409
    ParcialPATCH /productos/{id}200404/409/422
    BorrarDELETE /productos/{id}204404
    Nivel 0

    RPC: un endpoint y acciones en body.

    Nivel 1

    Recursos con URI, pero semantica pobre.

    Nivel 2

    Verbos + status codes. Objetivo realista.

    Nivel 3

    HATEOAS: respuestas con enlaces de navegacion.

    Produccion

    Consistencia, versionado, errores y documentacion.

    8. Statelessness y cache HTTP

    Stateless significa que cada request trae todo lo necesario. La cache evita repetir trabajo cuando el recurso no cambio.

    ETag / If-None-Match

    Cliente -> GET /libros/1
Servidor -> 200 OK
            ETag: "v3"
            body completo

Cliente -> GET /libros/1
           If-None-Match: "v3"
Servidor -> 304 Not Modified
            sin body

    Cache-Control

    max-age=3600 permite usar copia una hora. no-cache obliga a revalidar. no-store prohibe guardar, util para datos sensibles.

    Escalado

    Si el servidor no guarda sesion, cualquier replica puede atender cualquier request. El token viaja en cada llamada.

    Sesion vs token

    Una cookie de sesion puede ser stateful si apunta a memoria del servidor. Un JWT suele encajar mejor con stateless, aunque no sustituye buen diseno de seguridad.

    304

    El 304 no trae body. Su mensaje real es: "tu copia sigue vigente".
