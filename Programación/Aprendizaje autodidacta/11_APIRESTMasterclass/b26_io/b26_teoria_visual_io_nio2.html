Dos mundos que conviven

java.io bytes
InputStream y OutputStream: flujo secuencial de bytes. Es la base para ficheros, sockets, memoria y filtros.
java.io texto
Reader y Writer: flujo secuencial de caracteres. Necesitan charset cuando se conectan a bytes.
objetos
ObjectInputStream y ObjectOutputStream: convierten grafos de objetos en bytes y los reconstruyen.
java.nio.file
Path y Files: rutas y operaciones de alto nivel. Es la opción preferida para código nuevo.
canales
FileChannel y ByteBuffer: control fino, buffers, posiciones, transferencias eficientes y memory-mapped I/O.
Regla práctica: para código nuevo usa NIO.2 cuando puedas. Conoce java.io porque aparece en APIs antiguas, sockets, serialización y en el examen.

26.1 · Flujos de bytes

InputStream / OutputStream

La unidad mínima de E/S es el byte. Un InputStream lee bytes desde una fuente; un OutputStream escribe bytes hacia un destino.

Jerarquía práctica

InputStreamcontrato de lectura
→
FileInputStreamlee bytes de fichero
OutputStreamcontrato de escritura
→
FileOutputStreamescribe bytes a fichero
FilterInputStreamenvuelve otro stream
→
BufferedInputStreamreduce llamadas al SO
ByteArrayOutputStreamdestino en memoria
→
byte[]resultado acumulado
Copia canónica con buffer
try (InputStream in = new FileInputStream(origen);
     OutputStream out = new FileOutputStream(destino)) {
    byte[] buf = new byte[8192];
    int n;
    while ((n = in.read(buf)) != -1) {   // read devuelve bytes leídos o -1 en EOF
        out.write(buf, 0, n);            // escribe solo n bytes, no el array entero
    }
}
Tres detalles de examen: read() devuelve un int de 0 a 255 o -1; el último buffer puede ir medio lleno; envolver en BufferedInputStream/BufferedOutputStream mejora rendimiento.

26.3 · Acceso aleatorio

RandomAccessFile: el fichero como array en disco

Los streams normales son secuenciales: avanzan de principio a fin. RandomAccessFile permite saltar a una posición exacta con seek(offset) y leer o escribir ahí.

Modelo mental: hay un puntero de fichero. Cada lectura o escritura lo mueve. seek lo recoloca manualmente.
Leer y sobrescribir en una posición
try (RandomAccessFile raf = new RandomAccessFile(f, "rw")) {
    raf.write(new byte[]{10, 20, 30, 40});
    raf.seek(2);                 // salta al byte de índice 2
    int b = raf.read();          // 30
    raf.seek(2);
    raf.write(99);               // sobrescribe en sitio
    long tam = raf.length();
}

Registros de tamaño fijo

Si cada registro mide lo mismo, puedes saltar directamente al k-ésimo registro sin leer los anteriores.

posición = índice * tamañoRegistro

Por eso writeInt/readInt, writeDouble/readDouble o campos con longitud pactada importan: un int siempre ocupa 4 bytes, mientras que writeUTF tiene tamaño variable.

  • getFilePointer() te dice dónde está el puntero.
  • Modos: r, rw, rws (datos y metadatos síncronos), rwd (datos síncronos).
  • setLength() trunca o extiende.
  • Al extender, el contenido de la nueva zona queda sin definir hasta que lo escribas: no presupongas que son ceros.

Lo practicas en Ej209RandomAccessFile: offsets, actualización en sitio, truncar/extender, registros fijos y recorrido inverso.

26.4 · Serialización

ObjectOutputStream convierte objetos en bytes

Serializar es transformar un objeto y su grafo de referencias en bytes. Deserializar es reconstruirlo. La clase debe implementar Serializable, una interfaz marcadora sin métodos.

Viaje de un objeto

Objeto vivocampos normales + transient
→
writeObject()transient se omite
Bytes serializadosfichero, memoria, red
→
readObject()objeto reconstruido
Serializar y reconstruir
try (ObjectOutputStream oos = new ObjectOutputStream(new FileOutputStream(f))) {
    oos.writeObject(new Punto(3, 4));    // Punto implements Serializable
}
try (ObjectInputStream ois = new ObjectInputStream(new FileInputStream(f))) {
    Punto p = (Punto) ois.readObject();  // puede lanzar ClassNotFoundException
}
transient
El campo no se serializa. Al reconstruir vuelve a null, 0 o valor por defecto. Útil para secretos, cachés o datos derivados.
serialVersionUID
Identifica la versión compatible de la clase. Si no coincide, puede saltar InvalidClassException. Conviene declararlo explícito.
Grafo en cascada
writeObject sigue referencias. Si dentro hay objetos no serializables, puede aparecer NotSerializableException.
Seguridad
No deserialices datos no confiables. Para datos externos suele preferirse JSON, Protobuf u otro formato controlado.

Lo practicas en Ej210ObjectSerialization: records, colecciones, transient, grafos, clonación por serialización y NotSerializableException.

26.5 · NIO.2

Path y Files: rutas y operaciones modernas

Path representa una ruta, no un fichero abierto. Files concentra operaciones habituales: leer, escribir, copiar, mover, borrar, crear directorios, consultar tamaño.

Operaciones frecuentes en una línea

Files.writeString(p, "hola");

Files.copy(origen, destino, StandardCopyOption.REPLACE_EXISTING);
Files.move(origen, destino, StandardCopyOption.REPLACE_EXISTING);
Files.deleteIfExists(p);

26.6 · Árboles y atributos

Recorrer directorios también abre recursos

Files.walk, find, list y lines devuelven streams que deben cerrarse. Son Streams de Java, pero respaldados por recursos del sistema.

Recorrido y filtrado
try (Stream<Path> s = Files.walk(dir)) {
    long ficheros = s.filter(Files::isRegularFile).count();
}
try (Stream<Path> s = Files.find(dir, 5,
        (path, attrs) -> path.toString().endsWith(".txt"))) {
    // recorrer + filtrar con atributos
}
try (Stream<Path> s = Files.list(dir)) {
    // solo hijos directos, no recursivo
}
Files.walk(dir)
Recorre todo el subárbol. Puede limitarse con profundidad: walk(dir, profundidad).
Files.find(dir, max, pred)
Recorre y filtra usando ruta y atributos en la misma operación.
Files.list(dir)
Lista solo los hijos directos.
Files.mismatch(a,b)
Devuelve -1 si los ficheros son iguales o la posición del primer byte distinto.
Atributos básicos
BasicFileAttributes a = Files.readAttributes(p, BasicFileAttributes.class);
a.isRegularFile();
a.size();
a.creationTime();
a.lastModifiedTime();

26.7 · Temporales, canales y buffers

FileChannel y ByteBuffer: más cerca del hardware

NIO también trabaja con canales y buffers. Un FileChannel lee o escribe contra un ByteBuffer, que tiene capacidad, posición y límite.

Escribir y leer por ventanas con un canal
try (FileChannel ch = FileChannel.open(p, CREATE, WRITE, TRUNCATE_EXISTING)) {
    ByteBuffer src = ByteBuffer.wrap(datos);
    while (src.hasRemaining()) {
        ch.write(src);                 // una escritura puede ser parcial
    }
}
try (FileChannel ch = FileChannel.open(p, READ)) {
    ByteBuffer buf = ByteBuffer.allocate(8192);
    while (ch.read(buf) != -1) {
        buf.flip();                    // escritura del canal → lectura del programa
        while (buf.hasRemaining()) {
            byte actual = buf.get();   // procesar incrementalmente
        }
        buf.clear();                   // reutilizar; no borra los bytes antiguos
    }
}
Por qué este patrón escala: no convierte ch.size() a int ni carga el fichero entero. Además, explicita CREATE/TRUNCATE_EXISTING y admite transferencias parciales.

Simulador de ByteBuffer

Mira cómo cambian position y limit. Después de llenar el buffer, flip() pone position a 0 y limit donde estaba la posición.

capacity
Tamaño total reservado por el buffer.
position
Índice donde ocurrirá la próxima lectura o escritura.
limit
Frontera hasta la que puedes leer o escribir.
remaining()
limit - position. Indica cuántos bytes quedan por procesar.
clear()
Prepara para reutilizar. No borra los datos antiguos.
compact()
Conserva al principio los bytes todavía no consumidos y deja espacio para recibir más.
  • Files.createTempFile y createTempDirectory crean temporales en java.io.tmpdir.
  • ch.map(READ_ONLY, 0, size) mapea una región a memoria, útil para grandes índices y acceso aleatorio.
  • transferTo/transferFrom pueden copiar canal a canal con ayuda eficiente del sistema operativo.

Lo practicas en Ej213TempFilesAndChannels: temporales, flip, capacidad/posición/remaining, mapeo, truncado y transferencias.

26.8 · Conversión entre formatos

Convertir es cambiar representación

Los datos viven como bytes, texto, properties, CSV, hex, Base64 o números en binario. Convertir significa leer una representación, transformar en memoria y escribir otra.

Texto, properties, CSV, números, Base64 y hex
// texto <-> binario
byte[] bytes = texto.getBytes(StandardCharsets.UTF_8);
String vuelta = new String(bytes, StandardCharsets.UTF_8);

// .properties
Properties props = new Properties();
props.setProperty("nombre", "ana");
try (OutputStream os = new FileOutputStream(f)) { props.store(os, null); }
try (InputStream is = new FileInputStream(f)) { props.load(is); }
String v = props.getProperty("nombre", "N/A");

// CSV manual
String[] campos = linea.split(",", 2);
String csv = String.join(",", List.of("a", "b", "c"));

// número <-> bytes, tamaño fijo, big-endian
byte[] b = ByteBuffer.allocate(4).putInt(123456).array();
int n = ByteBuffer.wrap(b).getInt();

// bytes <-> texto transportable, no cifrado
String b64 = Base64.getEncoder().encodeToString(bytes);
String hex = HexFormat.of().formatHex(bytes);
Detalle de Properties: las variantes con InputStream/OutputStream usan ISO-8859-1 y escapes Unicode por compatibilidad histórica. Si el formato debe ser UTF-8 explícito, usa load(Reader)/store(Writer,...).
Ojo con CSV: split(",") no maneja comas dentro de comillas. Para CSV real necesitas parser. Y Base64/hex no cifran ni comprimen: solo convierten bytes a texto ASCII portable.

Lo practicas en Ej214FormatConversion: texto ↔ binario, properties ↔ CSV, número ↔ bytes, Base64 y hex.

Modelo mental transversal

El viaje de un dato: una sola historia para todo el bloque

Las clases cambian, pero el recorrido casi siempre es el mismo. Primero identifica la representación del dato y el tipo de acceso; después elige la abstracción más alta que resuelva el problema.

01
Representación
¿Qué existe realmente? Bytes, caracteres, objetos o registros. Un fichero nunca “sabe” que contiene texto: solo conserva bytes.
02
Dirección
Entrada o salida. Lees desde una fuente hacia el programa; escribes desde el programa hacia un destino.
03
Acceso
Secuencial, aleatorio o por árbol. El patrón de acceso decide entre stream, RandomAccessFile, canal o recorrido NIO.2.
04
Transformación
Charset, serialización o formato. Aquí los bytes adquieren significado: texto UTF-8, entero big-endian, objeto, Base64, hex o CSV.
05
Ciclo de vida
Abrir, procesar, cerrar. El recurso pertenece a un try-with-resources; el cierre es parte del algoritmo, no una tarea de limpieza opcional.

Selector de herramienta

Elige la forma del problema. El resultado muestra la API, el motivo y la trampa que debes vigilar.

Path

Files

Texto completo de tamaño razonable

Files.readString y Files.writeString expresan directamente la intención. Fija UTF-8 cuando quieras que el contrato sea explícito.

Vigila: cargar todo el contenido en memoria no es apropiado para ficheros enormes.

Piezas concretas que completan el mapa

ByteArrayInputStream
ByteArrayOutputStream
Aplican el mismo contrato de streams sobre memoria. Son útiles para preparar, probar o transformar un byte[] sin tocar disco; la salida se recupera con toByteArray().
FilterInputStream
FilterOutputStream
Son la base del patrón decorador: envuelven otro stream para añadir comportamiento. BufferedInputStream y BufferedOutputStream reducen llamadas al sistema.
readInt()
readDouble()
readUTF()
En RandomAccessFile, los primitivos numéricos usan big-endian y tamaño fijo; readUTF/writeUTF usan longitud + UTF modificado y por eso el campo es variable. readUnsignedByte() evita interpretar 128–255 como negativos.
getFileName()
normalize()
Path.getFileName() devuelve el último componente. normalize() simplifica elementos ./.. de forma léxica; no abre el fichero ni equivale a toRealPath().
Files.write(..., APPEND)
FileTime
Files.write(p, List.of("línea"), StandardOpenOption.CREATE, StandardOpenOption.APPEND) añade líneas. Las fechas de BasicFileAttributes son valores FileTime.
serialVersionUID
filtro de entrada
Decláralo con intención: private static final long serialVersionUID = 1L;. Un ObjectInputFilter reduce superficie, pero no convierte en segura una fuente no confiable ni elimina el riesgo de gadget chains.
Jerarquía de decisión: empieza por Path/Files; usa Reader/Writer si procesas texto en streaming, streams si necesitas bytes y canales cuando necesitas buffers, posiciones o transferencias de bajo nivel.
Atlas visual interactivo

Ver el estado interno para dejar de memorizar a ciegas

Estas cuatro lentes conectan los temas más abstractos: posición física, alcance de un recorrido, memoria consumida y cambio de representación.

1 · RandomAccessFile: el offset es una coordenada

Imagina seis registros de 12 bytes. Mover el selector no recorre los anteriores: calcula directamente offset = índice × 12 y coloca allí el puntero.

offset = 24 bytes

Matiz: writeInt y writeDouble tienen tamaño fijo; writeUTF añade una longitud y usa UTF modificado, por lo que su tamaño depende del texto. Si extiendes el fichero, el contenido de la nueva zona no está definido hasta escribirlo.

2 · Un árbol, tres operaciones distintas

list mira solo los hijos directos; walk recorre el subárbol; find recorre y filtra con la ruta y sus atributos. Prueba los tres enfoques sobre el mismo árbol.

Los streams devueltos por list, walk, find y lines mantienen recursos abiertos: envuélvelos en try-with-resources.

Round-trip: si conservas charset y algoritmo, puedes volver al dato original. Hex usa dos caracteres por byte; Base64 ronda cuatro caracteres por cada tres bytes. Ninguno cifra ni comprime.

.properties tiene una excepción histórica: load(InputStream)/store(OutputStream) usan ISO-8859-1 y escapes Unicode. Para controlar UTF-8, usa las sobrecargas con Reader/Writer.

Precisiones profesionales para Java 21

1
No confundas comprobar con garantizar

Files.exists(p) sirve como consulta, pero el fichero puede cambiar antes de la siguiente operación. Intenta la operación y trata su excepción.

2
Una llamada no siempre vacía el buffer

Con canales, itera mientras haya datos pendientes. Para lectura continua: read → flip → get → clear/compact.

3
clear() no borra

Restablece position=0 y limit=capacity. compact() conserva los bytes todavía no consumidos.

4
Un grafo cíclico puede serializarse

La serialización conserva referencias compartidas y ciclos. El problema aparece si alcanza un objeto no serializable, crece sin control o proviene de una fuente no confiable.

5
transient no cifra

Solo omite el campo en la serialización por defecto; no sustituye un diseño de seguridad ni impide que código personalizado lo escriba.

6
Las utilidades tienen versiones distintas

NIO.2 llegó en Java 7; Path.of/readString en Java 11, mismatch en Java 12 y HexFormat en Java 17. Este módulo compila con Java 21.
